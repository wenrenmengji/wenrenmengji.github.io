    第一章：版本控制系统（VCSs）和Git的基本概念

    • 版本控制系统首先是一种系统，它可以记录一个或多个文件内容变化，使得将来可以查阅特定版本的修改情况。
    • Git直接记录快照，而非差异比较、本地执行、完整性、一般仅添加数据。
      三种状态：committed（已提交）、modified（已修改）、staged（以暂存）。
      三个阶段：工作区（克隆到本地）、暂存区（二进制索引文件）、Git目录（远程仓库）。
      使用方式：命令行（推荐）、GUI。
      配置：仓库级、用户级、系统级。低级别配置会覆盖高级别配置
      用户级的优先级高于系统级的优先级，所以用户级配置会覆盖系统级别，比如系统级

      设为 master， 但用户级设为 main。最终生效值：main（用户级优先级更高）。

    第三章：Git的分支

    • 生动的比喻

      -在看到一个简单的提交历史示意图时,我联想到了一个画面:一个 火车工作站在铁路上铺铁轨,工 作站相当于当前分支所指的提交。

      -你的联想非常生动！Git 提交历史确实可以比作 铁路铺设系统，而 分支和提交 就像 铁轨和工作站 的动态扩展过程。我们来拆解这个比喻，让它更贴合 Git 的实际运作逻辑：

      🚂 Git 铁路模型详解

      1. 铁轨 = 提交历史（Commit Chain）

      • 每段铁轨：代表一个提交（commit），用哈希值（如 c842e05）唯一标识。
      • 铁轨方向：从旧提交指向新提交（就像时间不可逆，Git 历史默认向前推进）。

      2. 工作站 = 分支指针（Branch Pointer）

      • 工作站位置：分支（如 main、feature）的指针指向某段铁轨（某个提交），表示 当前的工作基点。
      • 动态移动：
        • 当你 git commit，工作站会向前铺设一段新铁轨，并移动到这个新节点。
        • 当你 git merge，相当于从另一条轨道合并铁轨到当前路径。

      3. 道岔 = 分支分叉（Branching）

      • 分叉点：两个分支从同一个提交点分开（比如从 main 创建 feature 分支）。
      • 并行施工：不同工作站（分支）可以在各自的轨道上独立铺设铁轨（提交代码）。

      🛠️ 关键操作对应现实场景

      Git 操作 铁路施工场景
      git commit 从当前工作站铺设一段新铁轨，并移动工作站到新位置。
      git checkout -b feat 在当前铁轨位置分叉出一条新支线，并派一个工作站去支线施工。
      git merge feat 将支线的铁轨连接回主线，可能需解决冲突（调整轨道接口）。
      git rebase main 把支线的铁轨拆下来，重新接到主线的尖端（保持线性历史）。
      git reset --hard 直接拆掉当前工作站之后的铁轨（谨慎使用！）。

      ⚠️ 注意事项

      1. 不可逆性：
        • 铁轨一旦铺设（提交），默认无法修改（除非用 git rebase 或 reset 等“施工事故工具”）。
      2. 冲突 = 轨道接口不匹配：
        • 当两条分支修改了同一段代码，合并时需要手动调整接口（解决冲突）。
      3. HEAD = 移动中的工作站：
        • HEAD 指向你当前所在的工作站（分支指针的位置）。

      🚀 用铁路模型理解 Git 核心

      • 分支是轻量的：创建分支只是新增一个可移动的工作站指针（不复制铁轨）。
      • 提交是安全的：每次提交都会生成唯一的铁轨段，历史不会被意外破坏。
      • 远程仓库 = 中央铁路枢纽：git push 是将本地铁轨同步到中央网络。

      下次操作 Git 时，可以想象自己是个铁路工程师，用分支和提交铺设代码世界的轨道！ 🌉

      可以把工作从开发主线上分离，以免影响主线。

      HEAD：是一个特殊指针，指向当前所在的本地分支（可以将其想象为当前分支的别名）。

    • 分支管理

    • 分支开发工作流（分支策略）

      • 长期分支（可以理解为一种稳定分支，是一种主分支，其他的分支当达到稳定状态后会合并到这条分支上。）
      • 短期分支（可以叫主题分支，被用来实现单一特性或者相关工作。

    • 远程分支

    • 变基

      • 意思是重新整理提交历史，可以将一个分支的提交移植到另一个分支的最新提交之后，形成一条线性的历史记录。
      • 变基使得提交历史更加整洁，你在查看经过变基的分支的历史记录时会发现，尽管实际的开发工作时并行的，但他们看上去就像是串行的一样，提交历史时一条直线没有分叉。
      • ！！！如果提交存在于你的仓库之外，而别人可能基于这些提交进行开发，那么不要执行变基。

      • 快进合并（直接移动分支指针（不生成额外的合并提交哈希值），保持提交历史的线性。可以理解为指针跳跃。

      • 用变基解决变基

      • 合并 VS 变基

        总的原则是，只对尚未推送或分享给别人的本地修改执行变基操作清理历史， 从不对已推送至别处的提交执行变基操作

    第四章：服务器端的Git

    裸仓库（版本库的副本,但没有当前工作目录的仓库。就是你的工程目录内的 .git 子目录内容，存放的 

            只有Git 的资料，不包含其他资料。（**只有 Git 元数据，没有工作目录中的源码文件。**）

       习惯上(按照惯例)，裸仓库的目录名以 .git 结尾

 适合做“服务器端仓库”

    Git 的数据是以基本的静态文件形式提供的

    Git 服务器（小型的GitHub，让其他人可以通过网络访问、clone、push、pull代码。

    • 协议

      • 本地协议（local protocol）：直接通过本地文件系统路径访问 Git 仓库的简单方式，无需网 络协议。适用于同一台机器上的仓库克隆或同步，速度快。

        意味着你直接通过 文件系统路径（而非网络协议如 HTTP/SSH）来访问和操作 Git 仓库。这种方式的核心是 绕过网络层，直接在本地磁盘或共享存储中读写仓库数据。

      • HTTP 协议

        智能 HTTP 协议、非智能 HTTP 协议

      • SSH 协议

        一个验证授权的网络协议。

      • Git 协议

        守护进程

    • 在服务器搭建 Git

      首先在开始架设 Git 服务器前，需要把现有仓库导出为裸仓库。

      然后把裸仓库放到服务器上并设置你的协议。（能控制的服务器：本地电脑、云服务器（阿里云、腾讯云）、局域网服务器）

    • GitLab

      一个基于 Git 的一体化DevOps平台，提供从代码托管到项目管理等全生命周期工具。

      • 关于DveOps

        是一种软件开发方法论，强调开发（Dev）和运维（Ops）的协作，通过自动工具链实现，

        • 持续集成（CI）：开发者频繁提交代码到共享仓库，自动触发构建与测试
        • 持续部署（CD）：自动化部署到测试/生产环境
        • 监控与日志：实时监控应用性能
